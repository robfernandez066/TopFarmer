TASK-019: Add the data-driven FarmPlot crop lifecycle
Status: ready
Complexity: complex
Files: modify res://scripts/farm/farm_plot.gd; create res://tests/farm_plot_crop_lifecycle_test.gd and its Godot UID sidecar
Acceptance:
- Extend each FarmPlot from visual-only state into one deterministic crop lifecycle with exactly the existing `EMPTY`, `GROWING`, and `READY` states; one plot holds at most one crop ID and its two absolute integer UTC timestamps at a time
- Expose `plant_crop(crop_id: StringName, started_at_utc: int) -> bool`, `update_growth(now_utc: int) -> void`, `harvest_crop() -> StringName`, and `replant_last_crop(started_at_utc: int) -> bool` as the only new gameplay mutation methods; expose exactly `get_current_crop_id() -> StringName`, `get_last_harvested_crop_id() -> StringName`, `get_started_at_utc() -> int`, `get_ready_at_utc() -> int`, and `get_lifecycle_state() -> int` as read-only queries
- `plant_crop` succeeds only from EMPTY with a nonempty crop ID found through the read-only `Balance.get_crop` API and a nonnegative supplied UTC timestamp. It stores the crop ID and `started_at_utc`, reads `growth_sec_eff` from that crop row, computes `ready_at_utc = started_at_utc + ceil(growth_sec_eff)`, enters GROWING, and displays the existing GROWING visual
- No growth duration, crop yield, reward, price, or other gameplay number is hardcoded. Missing, nonnumeric, nonpositive, nonfinite, or out-of-range `growth_sec_eff` fails planting loudly with a developer diagnostic and leaves the plot byte-for-byte equivalent to its prior EMPTY lifecycle values and visual state
- Planting while GROWING or READY, planting an unknown or empty crop ID, or supplying a negative timestamp returns `false` without changing crop ID, timestamps, last harvested crop ID, lifecycle state, node identity, or visuals
- `update_growth` changes GROWING to READY exactly when supplied `now_utc >= ready_at_utc`; a value one second earlier remains GROWING. A supplied time earlier than `started_at_utc` grants no progress, changes no stored value, and cannot move any state backward. EMPTY and READY ignore growth updates
- Store only `started_at_utc` and `ready_at_utc`, never remaining duration. Do not call the operating-system clock, create a Timer, use `await`, or advance from frame delta in this task; the caller supplies UTC so later platform clock-integrity and save work can remain outside FarmPlot
- Use empty `StringName()` for unset crop IDs and integer `-1` for unset timestamps. `harvest_crop` succeeds only from READY: it returns the harvested crop ID, copies that ID into `last_harvested_crop_id`, clears the current crop ID and both timestamps to those sentinels, enters EMPTY, and restores the existing EMPTY visual. When called from EMPTY or GROWING it returns an empty `StringName` and changes nothing
- `replant_last_crop` is an explicit player-action helper, never automatic. It succeeds only from EMPTY when `last_harvested_crop_id` is still a valid Balance crop and delegates to the same planting validation with the newly supplied timestamp. It returns `false` and changes nothing before the first successful harvest or while a crop occupies the plot
- Preserve the current empty soil, GROWING runtime pair and offset, READY placeholder runtime pair and offset, shadow ordering, node hierarchy and identity, editor-selectable preview behavior, and existing visual-state test. Lifecycle transitions must drive those exact existing visuals without modifying any PNG, scene, project setting, balance file, or import sidecar
- Do not add input handling, touch areas, mouse behavior, UI, inventory, seeds, costs, yields, rewards, XP, particles, sound, vibration, camera behavior, persistence, offline reconciliation, clock-integrity policy, or automatic replanting in this task
- The new headless test loads the real Balance autoload and FarmPlot scene and proves: initial EMPTY sentinels; rejection paths are mutation-free; Sunwheat planting uses the CSV `growth_sec_eff` rather than a test literal; one-second-before and exact-ready boundaries; backward supplied time; occupancy rejection; premature harvest rejection; successful harvest and remembered crop; explicit one-action replant at a new timestamp; no offline auto-replant; and at least two complete plant/grow/harvest/replant cycles with exact visuals and no node drift
- The test also proves another valid crop row can be planted through the same generic API, an unknown crop cannot, current and last crop IDs remain distinct at the documented moments, and all returned booleans and harvested IDs match the contract
