TASK-022: Add a portable monotonic in-session UTC clock
Status: ready
Complexity: complex
Files: create res://scripts/platform/session_utc_clock.gd and its Godot UID sidecar; create res://tests/session_utc_clock_test.gd and its Godot UID sidecar
Acceptance:
- Create `class_name SessionUtcClock extends RefCounted` under `res://scripts/platform/`; it uses only Godot's cross-platform `Time.get_unix_time_from_system()` and `Time.get_ticks_msec()` APIs and contains no Android, Java, network, account, cloud, save, scene-tree, frame-delta, or OS-specific code
- Expose exactly `start() -> bool`, `now_utc() -> int`, and `is_started() -> bool`. Before a successful start, `is_started` is false and `now_utc` returns `-1` with a developer diagnostic
- On the first successful `start`, sample system UTC exactly once, require a finite nonnegative value that fits the signed 64-bit timestamp calculation, round it down to an integer second, sample a nonnegative monotonic millisecond value exactly once, and store both as immutable session baselines; return `true` and set `is_started` true
- A failed `start` leaves the clock fully unstarted and may be retried. Calling `start` after success returns `false`, emits a developer diagnostic, and cannot resample or alter either baseline or the last returned UTC value
- After start, `now_utc` never reads system UTC again. It computes `baseline_utc + floor((current_monotonic_msec - baseline_monotonic_msec) / 1000)` from the original baselines, returning an integer absolute UTC Unix timestamp; the `1000` conversion is a time-unit conversion, not a gameplay duration
- Sub-second monotonic advances return the same UTC second; reaching each exact 1000-millisecond boundary advances one second. Computation always uses total elapsed time from the immutable baseline rather than incrementally adding frame deltas, so call frequency cannot introduce drift
- Track the last observed monotonic value and last returned UTC. If a later monotonic sample is invalid, negative, lower than the baseline, or lower than the prior observed sample, emit a developer diagnostic and return the last valid UTC without changing any stored baseline or moving time backward
- Detect signed 64-bit addition overflow before adding elapsed seconds. On overflow, emit a developer diagnostic and return the last valid UTC without changing state; no wraparound, negative timestamp, crash, or plot mutation is allowed
- Changes to the device wall clock after successful start cannot change `now_utc`, because the system UTC API is never sampled again during that clock instance. This task does not claim trusted-device-time verification and does not grant offline elapsed time
- Keep the two raw-read operations behind overridable internal methods so the first-party test can supply deterministic wall-clock and monotonic sequences without adding public test-only mutation methods or a third-party dependency
- Prove that `Callable(clock, "now_utc")` is valid after start and can configure the existing FarmPlotInteractionController. In the test, a deterministic clock subclass advances a real FarmPlot from GROWING to READY at the exact CSV-derived deadline through controller process updates without the controller or plot reading an operating-system clock
- The headless test proves pre-start behavior, failed-start retry, exactly one wall-clock baseline read, exactly one monotonic baseline read, second-start immutability, sub-second and exact-second boundaries, large call-frequency independence, multiple-second jumps, post-start wall-clock-change immunity, monotonic regression clamping, invalid monotonic input, overflow protection, nondecreasing return values, callable/controller compatibility, exact data-driven crop readiness, and no automatic replant
- Do not register an autoload, modify `project.godot`, connect this clock to the main scene, edit the interaction preview, add pause/resume handling, persistence, offline reconciliation, clock-integrity notices, user-facing text, balance values, input, inventory, rewards, UI, or assets in this task
